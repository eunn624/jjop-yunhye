# 쩝쩝윤혜 (먹숭이) — 사이트 정의서

디자인 도구(Supernova 등)에 핸드오프하기 위한 화면·콘텐츠·기능 정리. 가게 데이터처럼 동적인 부분은 제외하고, 정적 UI 구조와 사용자가 보는 모든 텍스트·상태·인터랙션 위주로 정리했다.

---

## 1. 프로젝트 개요

- **이름**: 쩝쩝윤혜 (UI 상 마스코트 이름 "먹숭이")
- **목적**: 넥슨 판교 직원이 회식·팀 점심 가게를 동료 추천 기반으로 빠르게 찾는 사내 웹앱
- **타깃**: 사내 직원 (모바일·데스크탑 둘 다)
- **톤**:
  - 캐주얼 + 친근. 마스코트 캐릭터(원숭이 "먹숭이")가 화자.
  - 카피에 이모지/감탄사 적당히 (🍌, 🐒, "팀비", "법카").
  - 강한 컬러 톤: 노랑 (Primary `#FFC107`)
- **마스코트**: 먹숭이 — `cha-assets/` 폴더 안 이미지. 표정 5종(happy / hungry / sleepy / wink / full).

---

## 2. 정보 구조 (Information Architecture)

### 글로벌 네비게이션 (Top Nav)

좌측에 브랜드 로고(작은 마스코트 + "먹숭이" 워드마크), 가운데 4개 탭, 우측에 "넥슨 판교" 위치 chip + 유저 아바타.

| 탭 | 라벨 | 역할 |
|---|---|---|
| 1 | 추천 | 위저드로 조건 입력 → 매칭 결과 |
| 2 | 제보 피드 | 모든 가게 카드 그리드 (필터 가능) |
| 3 | 제보하기 | 새 가게 제보 폼 |
| 4 | 이번주 핫픽 | 좋아요 누적 순위 |

탑네비는 sticky. 항상 보임. 모바일에선 가운데 탭 영역 숨김(햄버거 메뉴 없음 — 단순화).

### "추천" 탭 내부 마이크로 라우터

```
landing → step1 → step2 → step3 → result
```

위저드는 4뎁스가 아니라 3개 스텝 (step1·2·3) + 랜딩 + 결과.

---

## 3. 화면별 상세

### 3.1 Landing (추천 탭 진입 시)

**레이아웃**: 좌우 2단 hero. 모바일에선 상하.

**좌측 (텍스트 영역)**:
- 아이브로우(작은 라벨): "조직운영비, 알차게 쓰자 🐒" (노랑 강조)
- 큰 헤드라인: "팀비 어디 쓰지<span class='accent'>?</span>"
  - 서브: "먹숭이가 동료 추천만 모았어요." (회색, 더 작은 폰트)
- 본문 카피 (3줄):
  > 법카 긁어본 사람들이 직접 남긴 한 줄 추천.<br/>
  > 팀장님도 만족하고, 팀비도 살아남는<br/>
  > **회식 한 끼**를 4가지 질문으로 찾아드려요.
- CTA 버튼 2개 (가로 배치):
  - Primary: "먹숭이한테 물어보기 →" → step1로
  - Ghost: "그냥 둘러볼래요" → 제보 피드 탭으로 이동
- 통계 풋터 (3개, 가로 배치):
  - ⏱ 평균 38초
  - · 등록 가게 {N}곳
  - · 이번주 신규 제보 {N}건

**우측 (마스코트 영역)**:
- 큰 마스코트 이미지 (full 표정, 400px)
- 위아래로 둥둥 떠있는 애니메이션 (`mascot-float`: 4초 ease-in-out, ±14px)
- 말풍선 2개:
  - 우상단: "오늘 어디 긁어? 🍌"
  - 좌하단: "법카 챙겼지? 💳"
  - 둘 다 살짝 출렁이는 애니메이션 (`bubble-bob`: 3.2초, 두 번째는 1.4초 딜레이로 비동기)
- 마스코트는 컨테이너 좌측으로 20px 이동

**랜딩 풋터**:
- "동료들이 알려준 가게 {N}곳" (작은 대문자 라벨)
- 등록 가게 6개 chip 미리보기 (가게명 · 장르). 클릭 시 제보 피드로 이동

**빈 상태 (가게 0건)**: 풋터 자리에 EmptyState 컴포넌트.

### 3.2 Step 1: 식사 유형 (StepMode)

- Stepper (1/3, 라벨: 언제 · 인원 · 장르) — 현재 단계는 노랑 강조
- 마스코트 + 말풍선: "안녕! 먹숭이야. 먼저 — **언제 가는 거**예요?"
- H1: "팀 점심이에요, 본 회식이에요?"
- 부제: "예산 단위, 영업 시간, 술 가능 여부가 통째로 달라져요."
- 2개 카드 옵션 (2열 그리드):
  - 🌞 **팀 점심 / 캐주얼** — "도보 10분 안, 회전 빠른 곳 · 메뉴 단가 부담 적음"
  - 🌙 **본 회식 / 법카 디너** — "술 가능, 룸·단체석 · 늦게까지 가능 · 2차 연계"
- 선택된 카드는 노랑 보더 + 연노랑 그라데이션 BG + 우상단 ✓ 뱃지
- 하단 버튼: "← 처음으로" (ghost) / "다음 →" (primary, disabled when no selection)

### 3.3 Step 2: 인원 (StepPeople)

- Stepper (2/3)
- 마스코트 + 말풍선: "{선택한 식사유형}이군요 🌞/🌙 — **몇 명** 가요?"
- H1: "총 몇 명?"
- 부제: "여러 개 선택 가능 · 비워두면 인원 무관으로 찾아드려요."
- 4개 카드 옵션 (2열, 다중 선택 가능):
  - 👥 **2~4명 소수팀** — "회의 전후 빠른 식사, 4인 테이블 한 개로 충분"
  - 🍻 **5~10명 팀회식** — "테이블 두 개 붙이기 / 작은 룸 정도"
  - 🎉 **11~20명 부서** — "룸·단체석 필수, 예약 권장"
  - 🏛️ **20명+ 대규모** — "한정식·중식 대형룸·고기집 위주"
- 선택된 카드 동일 강조 + ✓ 뱃지
- 하단: "← 처음으로" / "다음 →" (no selection도 통과 가능)

### 3.4 Step 3: 장르 + 예산 + 조건 (StepGenre)

- Stepper (3/3)
- 마스코트 + 말풍선: "마지막 — **음식 종류**는 끌리는 거 다 골라요. 안 골라도 돼요."
- H1: "땡기는 게 있어요?"
- 부제: "여러 개 선택 가능 · 비워두면 전체 추천"

**장르 그리드** (5열 × 2행, 다중 선택 chip-card):
| 장르 | 아이콘 |
|---|---|
| 한식 | 🍚 |
| 일식 | 🍣 |
| 중식 | 🥟 |
| 양식 | 🍝 |
| 아시안 | 🍜 |
| 고기구이 | 🥩 |
| 해산물 | 🦐 |
| 분식·면 | 🍱 |
| 튀김류 | 🍤 |
| 뷔페 | 🍽 |

**예산 (인당) chip 줄** (다중 선택, "여러 개 선택 가능" 힌트):
- ~1만원 / 1~2만원 / 2~3만원 / 3~5만원 / 5만원+

**추가 조건 chip 줄** (다중 선택):
- 도보 10분 이내 / 역 근처 / 주차 가능 / 예약 가능 / 룸/단체석 / 야외 테라스 / 술 가능 / 2차 추천 / 늦게까지 / 포장 가능 / 배달 가능

- 하단: "← 이전" / "먹숭이야 찾아줘 →" (primary)

### 3.5 Result (추천 결과)

**헤더**:
- 작은 마스코트 (56px, happy) + 우측에 두 줄 텍스트
  - 작은 노랑 라벨: "먹숭이의 추천 · {N}곳 매칭"
  - H2: 🌞/🌙 + 선택된 인원/장르/예산/조건 슬래시 결합 (최대 4개)
- 우측 상단: "← 조건 다시" (ghost, 작은 사이즈)

**라이브 필터 strip** (카드 형태):
- "장르:" 라벨 + GENRE_OPTS chip 줄
- "조건:" 라벨 + EXTRA_OPTS chip 줄
- 즉시 적용 (전체 위저드 안 거치고)

**카드 리스트** (세로 1열, **좋아요 desc → 제보 수 desc** 정렬):
- 카드 좌측: 썸네일 (200px 폭, 140px 높이)
- 카드 우측 텍스트:
  - 상위 3위: 🥇 No.1 / 🥈 No.2 / 🥉 No.3 라벨
  - No.1 옆에 "먹숭이's PICK" 노랑 뱃지
  - 가게명 (큰 폰트)
  - 메타 줄: 장르 · 인당예산 · 인원 · "룸 가능" 같은 부가
  - 한 줄 추천 (회색 박스): `"코멘트"` — 닉네임 · 팀 · 제보 N건
  - 특징 chip 줄 (최대 4개, ✓ {label})
- 카드 우측 끝 (세로 정렬):
  - ❤️ 좋아요 버튼 (카운트 포함, pill)
  - "상세 →" primary 버튼

**필터 미매칭 빈 상태**:
- 마스코트 sleepy (120px)
- H3 "조건에 맞는 가게가 없어요"
- 부제 "조건을 조금 풀어볼까요?"
- "필터 초기화" primary 버튼

**전체 0건**: EmptyState 컴포넌트 (3.10 참조)

**하단**: "필터 초기화" ghost 버튼 가운데 정렬

### 3.6 제보 피드 (ReportFeed)

**헤더**:
- H2: "제보 피드"
- 부제: "동료들이 직접 다녀와서 남긴 한 줄 추천이에요."

**필터 strip** (가게 있을 때만):
- 장르 chip 줄 + 조건 chip 줄
- 우측에 "필터 초기화" 텍스트 버튼 (활성 시만 노출)

**카드 그리드** (3열, 반응형: 900px↓ 2열, 600px↓ 1열, gap 24px):
- 카드 (세로 레이아웃):
  - 상단: 썸네일 140px 높이 라운드 8
  - 중단 헤더 줄:
    - 가게명 (큰 폰트, ellipsis)
    - 💬 N (댓글 수, 있을 때만)
    - ❤️ 좋아요 버튼 (작은 pill, 카운트 포함, 토글)
  - 메타 줄: 장르 · 예산 · 닉네임 (회색, ellipsis)
  - 특징 chip 줄 (최대 4개, 초과는 "+N"으로 축약)
  - 한 줄 추천 ("코멘트", 2줄 clamp)
- 카드 클릭 시 상세 모달

**필터 미매칭 빈 상태**: 마스코트 + "조건에 맞는 가게가 없어요" + 필터 초기화 버튼
**전체 0건**: EmptyState

### 3.7 제보하기 (ReportForm)

**상단**: 마스코트 hungry (128px, 큰 사이즈) + 말풍선
- "팀비 잘 쓰는 법, 동료들이 궁금해해요. **가게 정보**를 알려주세요."

**폼 필드** (위에서 아래로):

| 필드 | 타입 | 비고 |
|---|---|---|
| 가게 이름 * | text input | 중복 시 빨간 박스 경고 ("⚠️ {이름}은(는) 이미 등록되어 있어요") |
| 주소 * | text input | |
| 네이버 지도 URL (선택) | url input | 빈 칸이면 자동 매칭. 입력 시 그 링크 사용 |
| 점심 / 저녁 | chip 단일 선택 | 🌞 점심 / 🌙 저녁 / 둘 다 |
| 장르 | select | GENRE_OPTS 10개 |
| 분위기 | chip 단일 선택 | 조용히 먹고 끝내기 / 수다 위주 / 격식 있게 / 캐주얼 |
| 적정 인원 (여러 개 선택 가능) | chip 다중 | PEOPLE_OPTS 4개 |
| 인당 예산 (여러 개 선택 가능) | chip 다중 | PRICE_OPTS 5개 |
| 특징 (해당되는 것 모두) | chip 다중 | EXTRA_OPTS 11개 |
| 한 줄 추천 | textarea | placeholder: "예: 법카 긁기 딱 좋은 가격대. 팀장님도 만족하심." |
| 닉네임 | text input | placeholder: "예: 회식왕" |
| 팀 / 부서 | text input | placeholder: "예: 개발팀" |

**하단**: "제보 올리기" primary 버튼 (제출 중엔 "제출 중...")

성공 시 토스트: "먹숭이가 받았어요! 동료들 팀비 고민 해결에 큰 도움이 됩니다 🍌"

### 3.8 이번주 핫픽 (HotPicks)

**헤더**:
- H2: "🔥 이번주 핫픽"
- 부제: "좋아요를 많이 받은 가게들이에요."

**카드 리스트** (세로 1열, 좋아요 desc → 제보 수 desc):
- Result 카드와 유사하지만 노출 정보 살짝 다름:
  - **#N** 순위 (No.1/2/3 메달 표기 아님, 그냥 #1, #2, ...)
  - ❤️ N (좋아요 수, 분홍 텍스트)
  - 가게명 · 장르·서브 · 예산
  - 한 줄 추천 회색 박스
- (좋아요 버튼은 핫픽엔 없음 — 단순 표시. 토글하려면 카드 클릭으로 상세 들어가서)

**0건**: EmptyState

### 3.9 상세 모달 (DetailModal)

**모드 2개**: view (보기) / edit (수정 폼)

**우상단 액션 클러스터** (`.modal-actions`):
- 작은 원형 아이콘 버튼 (SVG, 34×34):
  - **수정** (pencil) — view 모드에서만 노출, 클릭 시 edit 모드
  - **삭제** (trash) — view 모드에서만, 빨간색 (.danger)
  - **닫기** (X) — 항상 노출

**view 모드**:
- 히어로 (200px 높이): 가게 썸네일 + 하단 어두운 그라데이션
  - 그라데이션 위 좌하단:
    - 작은 chip 줄 (어두운 배경): "점심"/"저녁"/"점심·저녁" · "룸" · "주차" · 네이버 카테고리
    - 큰 가게명 (24px, 흰색, 그림자)
    - 장르·서브 (작은 회색)
- 본문 (32-36px 패딩):
  - 📍 주소
  - 2칸 그리드 (회색 박스):
    - 인당 예산: {priceRange}
    - 인원: {people}
  - 특징 chip 줄 (✓ {label}, 노랑 톤)
  - 한 줄 추천 박스 (배경 베이지, 보더 베이지):
    - `"{comment}"` — {nickname} · {team}
  - 🗺 "네이버 지도에서 보기" primary 버튼 (full-width)
  - **댓글 섹션** (3.11 참조)

**edit 모드**:
- 헤더: "맛집 정보 수정"
- PlaceForm 컴포넌트 (3.7과 동일한 필드)
- 하단: "저장" primary / "취소" ghost

### 3.10 EmptyState (빈 상태 공용 컴포넌트)

여러 곳에서 재사용:
- 마스코트 sleepy (120px, 가운데 정렬)
- H3: "아직 아무도 제보 안 했어요."  (기본 문구, 페이지마다 커스텀 가능)
- 본문: "팀비 어디다 쓰는지 알려주세요" (기본)
- "제보하러 가기 →" primary 버튼

### 3.11 댓글 섹션 (CommentsSection)

상세 모달 하단 (`borderTop` 구분선).

**헤더**: "댓글 {N}" (회색 카운트)

**댓글 리스트** (있을 때, 시간 오름차순):
- 회색 박스 (`#f7f7f8`, radius 8, 패딩 10/12)
- 본문 (12-14px, line 1.5)
- 메타: 닉네임 · 팀 · 시간 (회색 11px)
  - 시간 표시: "방금" / "{N}분 전" / "{N}시간 전" / "{N}일 전" / "M/D"

## 9. 매칭 / 정렬 규칙

### 매칭 (위저드 → 결과 필터)

`filterPlaces(places, filters)` 적용 순서:
1. mealType: 가게가 "both"거나 user 선택과 일치
2. people 다중 선택: 선택된 인원 버킷 중 하나라도 가게 capMin~capMax 와 범위 겹침
3. genres 다중 선택: 가게 genre가 선택 중 하나
4. budget 다중 선택: 선택된 가격대 중 하나라도 가게 priceMin~priceMax 와 범위 겹침
5. conditions 다중 선택: 선택된 모든 조건이 가게 extras에 포함 (AND)

> mood 필터는 의도적으로 적용 안 함 (가게마다 mood 단일이라 너무 좁아짐).

### 정렬 (Result, HotPicks 둘 다)

1. 좋아요 누적 카운트 desc (서버 + 세션 optimistic)
2. 동률이면 reports desc
3. (HotPicks는 그 전에 sortByRecent로 안정 정렬 베이스)

상위 3위: 🥇/🥈/🥉 메달 표기 (Result만), No.1 옆 "먹숭이's PICK" 뱃지.

## 10. 핵심 사용자 흐름

### A) 추천받기
랜딩 → "먹숭이한테 물어보기" → Step1 (식사유형, 필수) → Step2 (인원, 선택) → Step3 (장르·예산·조건, 모두 선택) → Result.

### B) 둘러보기
랜딩 → "그냥 둘러볼래요" → 제보 피드 (필터로 좁혀가며 탐색).

### C) 제보하기
탑네비 "제보하기" → 폼 작성 → "제보 올리기" → 토스트 → Apps Script가 시트 + JSON 자동 커밋 → 다음 새로고침부터 노출.

### D) 좋아요
어디서든 ❤️ 버튼 → 즉시 카운트 +1 (optimistic) → 서버 누적 → 다음 로드 시 서버값 반영. 핫픽/Result 정렬에 반영.

### E) 댓글
상세 모달 → 댓글 입력 → "보내기" → 즉시 리스트에 추가 → 서버 시트 저장 → 다음 로드 시 동기화.

## 13. 알려진 한계 / 디자인 시 참고

- 좋아요 카운트는 서버 누적이지만 POST가 no-cors라 응답 못 읽음. UI는 optimistic delta로 즉시 반영. 다음 페이지 로드의 GET이 진실의 원본.
- 가게 데이터는 GitHub 커밋 워크플로우라 추가/수정 후 새로고침 한 번 더 필요할 수 있음.
- BGM은 브라우저 자동재생 정책으로 음소거 자동재생 → 첫 사용자 인터랙션에 음소거 해제.
- 댓글·좋아요 시트는 사용자 인증 없음 (사내 사이트라 익명 허용).
